⬡ DESIGN NOTES & ARCHITECTURAL INTENT

Multiplayer Timing

Every system built as local GameScene state will need rearchitecting when a server becomes the authority. Keep state well-separated now. Hotseat first (shared browser, no networking), then WebSocket server. Don't let networking block feature depth — a boring game with multiplayer is still boring.

Tooltip Contract

Every time a new unit type is added, tooltips must be wired in all locations: the Planet Units section of NodePanel, the Build Modal card, and any panel where the unit is referenced by name. This is a standing rule, not a one-off task.

Projectile Architecture

AsteroidManager is the template for any free-moving object. Missiles follow the same pattern: spawn on event, move each update(), check arrival/intercept, fire resolution event. Planetary Defense buildings are radius checks run in the same update loop.

Planetary Power Intent

Power should be a hard cap, not a soft penalty. A planet with 4 power slots forces a real choice: offense (factory), defense (planetary defense), economy (extractor), utility (miner). Upgrades cost power to run — not just to build. This is the core tension of the building system.

Research & Replayability

Research is the primary replayability driver. Each game, players follow different tech paths — making their fleet and economy composition unique. Research should gate building upgrade tiers as well as ship stat improvements, so the tech tree has both military and economic branches.

AI Strategy

Implement a lightweight AI (expand-nearest, attack-weakest) now to seal the game loop for solo testing. Do not over-invest in AI before the feature set is stable — every new system (research, missiles, power) changes optimal AI behaviour. Deep AI is a Phase C problem.
